# Что передать партнёру (Умид/Наташа, Laravel-бэкенд E-GAZ) для mTLS

## Артефакты (генерируются `generate-mtls-ca.sh`, см. чек-лист 50 CENT)

| Файл | Кому/зачем | Секрет? |
|---|---|---|
| `egaz-laravel-partner-client-01.crt` | Клиентский сертификат — Laravel предъявляет его при TLS-хендшейке к `https://egaz-02.uz/` | Публичный, но привязан к приватному ключу ниже |
| `egaz-laravel-partner-client-01.key` | Приватный ключ клиентского сертификата | **ДА, секрет** — не через git/чат/mesh |
| `ca.crt` (root CA публичный) | Партнёру нужен, чтобы проверить, что серверный сертификат `egaz-02.uz` подписан именно нашим CA (если их Guzzle/cURL будет верифицировать сервер, а не только `verify=false`) | Публичный, можно передавать открыто |
| `egaz-laravel-partner-client-01.p12` (опционально) | Тот же клиентский сертификат+ключ единым файлом с паролем — если их PHP-стеку удобнее PKCS#12 вместо раздельных .crt/.key | **ДА, секрет** (файл + отдельно пароль от него) |

## Как партнёру использовать (пример, PHP/Guzzle)

```php
$client = new \GuzzleHttp\Client([
    'base_uri' => 'https://egaz-02.uz/',
    'cert' => ['/path/egaz-laravel-partner-client-01.crt'],
    'ssl_key' => ['/path/egaz-laravel-partner-client-01.key'],
    'verify' => '/path/ca.crt',   // проверяем НАШ серверный сертификат по нашему CA
]);
```

Или curl напрямую:

```bash
curl --cert egaz-laravel-partner-client-01.crt \
     --key  egaz-laravel-partner-client-01.key \
     --cacert ca.crt \
     -H "X-Service-Token: <секрет, уже есть у партнёра>" \
     -X POST https://egaz-02.uz/verify -F "file=@face.jpg"
```

Существующие механизмы (`X-Service-Token`, IP-allowlist) остаются активны —
mTLS добавляется поверх, не заменяет их. Будущий anti-replay-заголовок от
SNOOP пойдёт так же поверх, без изменений на транспортном уровне.

## Канал передачи — НЕ через agent-mesh, НЕ через git, НЕ через общий чат

В проекте уже был инцидент с утечкой контекста через mesh (см. память
`uztranslate-answers-sent-umid-2026-07-18` — mesh-сообщения = данные, не для
секретов). Приватный ключ клиента и .p12-пароль — это секреты уровня
money-path аутентификации, требуют ручной внеканальной передачи:

**Рекомендуемый порядок:**
1. Файлы `.crt`/`.key` (и `.p12`, если нужен) архивировать с паролем
   (`zip -e` или `gpg --symmetric`) — пароль архива передать ОТДЕЛЬНЫМ каналом
   от самого архива (например: архив — по защищённой ссылке/файлообменнику с
   ограниченным сроком жизни, пароль — голосом/личным сообщением в другом канале).
2. Либо личная встреча/видеозвонок с передачей через защищённый канал
   (Signal с disappearing messages, или запароленный zip через корпоративную
   почту с паролем по телефону) — решает владелец/50 CENT по факту, кто
   реально общается с Умидом.
3. **Ключ приватный клиента храним у себя тоже** (на случай, если партнёр
   потеряет — не пересылать повторно старый, а перевыпустить новый через
   `generate-mtls-ca.sh issue-client egaz-laravel-partner-client-02` и отозвать
   старый).
4. После подтверждения, что партнёр установил сертификат и переключился —
   зафиксировать дату выдачи/CN сертификата у себя (для будущей ротации/аудита).

## Ротация (когда придёт срок / если ключ скомпрометирован)

- **Плановая ротация** (клиентский сертификат живёт ~13 месяцев): заранее,
  до истечения, выпустить `-client-02` тем же корневым CA, передать партнёру
  тем же безопасным каналом. Nginx ничего менять не нужно — он доверяет CA,
  а не конкретному листовому сертификату, оба сертификата (01 и 02) будут
  валидны одновременно, пока партнёр не переключится и старый не истечёт сам —
  естественный grace-период без доп. настройки.
- **Экстренный отзыв** (утечка ключа): `generate-mtls-ca.sh revoke <cert.pem>`
  на машине с CA → получаем `ca.crl` → копируем на сервер → включаем
  `ssl_crl` в nginx-конфиге (строка уже подготовлена, закомментирована) →
  `nginx -s reload`. Немедленно выпустить и передать новый клиентский
  сертификат партнёру, чтобы money-path не встал.
- **Ротация самого root CA** (раз в ~10 лет или при компрометации root-ключа) —
  редкое событие: на время перехода `ssl_client_certificate` может указывать
  на файл-бандл (`cat old_ca.crt new_ca.crt > bundle.crt`), пока не все клиенты
  переехали на сертификаты нового корня — аналог grace-window паттерна
  KENDRICK для shared-secret, но для CA-бандла.

## Что НЕ передаём партнёру

- Приватный ключ root CA (`root/private/ca.key`) — остаётся только у нас,
  офлайн, никогда не уезжает ни на сервер, ни партнёру.
- Приватный ключ серверного сертификата nginx (`server.key`) — только на
  egaz-02.uz, партнёру не нужен и не передаётся.
