## 3. Фаза 0 — Фундамент ГСЧ (без функциональных изменений, самая быстрая, разблокирует остальное)

**Объём:** заменить `random` на `secrets.SystemRandom()` в `app/liveness_session.py::generate_challenge_spec` и в новом генераторе тайминга (Фаза 2). `secrets.SystemRandom` — это подкласс `random.Random` на `os.urandom`, drop-in по интерфейсу (`.sample`, `.randint` работают так же) — сигнатура функции не меняется, только источник энтропии.

**Зависимости:** нет, можно делать первым же коммитом.

**Риски:** практически нулевые — чистая замена ГСЧ, `rng: Optional[random.Random] = None` в сигнатуре уже допускает инъекцию (используется в тестах для детерминизма) — тесты не ломаются, продовый дефолт меняется.

**Кто:** SNOOP (backend).

**Критерий приёмки:** `python3 -c "from app.liveness_session import generate_challenge_spec; import inspect; ..."` подтверждает, что дефолтный `rng` — `secrets.SystemRandom`, не `random` (module-level). Существующие тесты `tests/test_liveness_endpoints.py` зелёные без изменений (детерминированные тесты продолжают инжектить свой `random.Random(seed)`).

## 5. Фаза 2 — Генерация: `secrets`-ГСЧ, случайный k=3-4, случайный порядок, рандомизация тайминга

### 5.1 Случайный выбор k и порядка

- `app/config.py`: заменить фиксированный `LIVENESS_CHALLENGE_STEP_COUNT: int = 2` на диапазон `LIVENESS_CHALLENGE_STEP_COUNT_MIN: int = 3` / `LIVENESS_CHALLENGE_STEP_COUNT_MAX: int = 4`.
- `app/liveness_session.py::generate_challenge_spec`: `k = rng.randint(step_count_min, step_count_max)`, затем `rng.sample(pool, k)` как сейчас (сэмпл уже даёт случайный порядок).

### 5.2 Расчёт итоговой энтропии (пул=6, k∈{3,4}, без учёта тайминга)

| k | Число упорядоченных выборок (nPr) | Формула |
|---|---|---|
| 3 | 120 | 6·5·4 |
| 4 | 360 | 6·5·4·3 |
| **Итого** | **480** | 120+360 |

`log2(480) ≈ 8.9 бит` против video-replay на уровне выбора шагов и их порядка — рост с текущего ~1 бита (при пуле=2, k=2, всегда обе перестановки) до ~8.9 бит. Это НЕ включает энтропию тайминга (см. ниже) — та добавляется поверх, но её вклад в битах не оцениваю численно без согласованного распределения окон (см. §5.3, отмечено как открытый вопрос, а не додуманное число).

Промежуточные пороги пула (для Волновой раскатки, Фаза 5) для сравнения:
- пул=3 (только +BLINK), k=2-3: nPr(3,2)+nPr(3,3) = 6+6 = 12 → `log2(12) ≈ 3.6 бит`.
- пул=5 (+NOD_UP/DOWN), k=3-4: nPr(5,3)+nPr(5,4) = 60+120 = 180 → `log2(180) ≈ 7.5 бит`.
- пул=6 (+SMILE), k=3-4: 480 → `≈ 8.9 бит` (целевое состояние спринта).

### 5.3 Рандомизация тайминга — новое поле контракта

Расширить `ChallengeSpec` (`app/main.py`) НЕ ломающе (только добавление поля, старые клиенты, читающие `steps`/`min_frames`/`max_frames`, не затронуты):

```
ChallengeSpec.step_windows: list[StepWindow]   # НОВОЕ, аддитивное поле
StepWindow: { step: str, min_delay_ms: int, max_delay_ms: int }  # окно ПОСЛЕ предыдущего шага/старта
```

Значения `min_delay_ms`/`max_delay_ms` сэмплируются тем же `secrets`-ГСЧ в момент генерации spec, из сконфигурированного диапазона (`LIVENESS_STEP_DELAY_MIN_MS`/`_MAX_MS`, значения — согласовать, не изобретаю сейчас, см. §9). Клиент (мобильное приложение) должен показывать инструкцию к шагу N не раньше `min_delay_ms` после показа шага N-1 — это межрепозиторийная зависимость на `egaz-mobile`/Android-клиент (Наташа/Умид), НЕ решается только этим сервисом.

**Открытый вопрос (см. §9):** точный численный диапазон окон тайминга — предмет согласования с Rustam/UX (те же 5-6с целевого флоу уже под риском по латентности CPU, см. `docs/LIVENESS_CONTRACT_v1.md` §4 п.3 — больше шагов и более широкие окна давят на тот же бюджет).

**Кто:** SNOOP (сервис) + координация с командой мобильного клиента через мост `egaz-mobile` (не наш код, зависимость).

**Риски:** межкомандная зависимость — тайминг-часть бесполезна, пока клиент не начнёт СОБЛЮДАТЬ окна на своей стороне (аналог уже пройденного паттерна `X-Request-Timestamp`: сервис готов раньше партнёра, включение поэтапное).

## 6. Фаза 3 — Серверная валидация первого контура

Три отдельных изменения, разного риска и разного режима раскатки.

### 6.1 Проверка ПОРЯДКА выполнения шагов (обязательная, без фиче-флага)

**Это прямое требование Рустама и главная дыра сегодня** — `verify_challenge` (`active_challenge.py:59-109`) ищет шаг по всей серии, без привязки к порядку. Меняем на: для шага `steps[i]` доказательство ищем только среди кадров с `seq > seq_последнего_доказательства_шага[i-1]` (монотонный указатель по возрастанию `seq`, `seq` есть всегда, не зависит от `captured_at`). Если для какого-то шага в разрешённом хвосте кадров доказательство не найдено — `STEP_NOT_DETECTED`, как сейчас, просто с более узким окном поиска.

**Это НЕ ломает контракт** (не новое обязательное поле, использует уже обязательный `seq`) — можно катить сразу, без флага, это чистое ужесточение уже существующей проверки `CHALLENGE_FAILED`/`STEP_NOT_DETECTED` семантики. Требует обновления `tests/test_liveness_endpoints.py` (кейсы, где шаги сегодня "проходят" в перепутанном порядке, должны начать падать).

### 6.2 Валидация `captured_at` (окно challenge + неубывание по `seq`)

`captured_at` сегодня `Optional[str] = None` — партнёр (Laravel) уже сделал M2-проверку у себя как ВТОРОЙ контур, наш — должен быть ПЕРВЫМ, значит появиться в этом сервисе тоже.

**Trade-off — ломаем ли контракт (обязательное поле) или мягкий режим:** выбираю **мягкий режим по тому же паттерну, что уже прижился в этом репозитории для `X-Request-Timestamp`** (`docs/LIVENESS_CONTRACT_v1.md` §6) — не делаю поле обязательным сразу:
1. Поле остаётся `Optional[str]` в схеме (не 422 для старых клиентов).
2. Новый флаг `LIVENESS_CAPTURED_AT_VALIDATION_ENABLED: bool = False` (`config.py`). Пока `False` — если `captured_at` присутствует, проверяем и **только логируем** аномалию (аудит-лог, не режем вердикт); если `None` — ничего не проверяем.
3. Партнёр (мобильный клиент через `egaz-mobile`) подтверждает, что стабильно шлёт `captured_at` на каждом кадре.
4. Только после подтверждения — флаг включается в `True`: `captured_at` отсутствующий или вне окна `[t_instruction_shown, expires_at]`, или убывающий по `seq`, начинает реально валить вердикт (`reason="CAPTURED_AT_INVALID"`, новый код в таблице `reason`).

Это сознательно НЕ то же самое, что "обязательное поле в Pydantic" — обход через мягкий rollout выбран, потому что репозиторий уже один раз прошёл ровно этот же trade-off (anti-replay timestamp) и он сработал без инцидента совместимости. Альтернатива (сделать поле обязательным немедленно) была бы быстрее для нас, но риск 422 на живом трафике партнёра без предупреждения — неприемлем по тому же прецеденту.

### 6.3 Window-timing по `t_instruction_shown` / `step_windows` (Фаза 2)

Тот же мягкий паттерн: новый флаг `LIVENESS_TIMING_VALIDATION_ENABLED: bool = False`, включается только после того, как клиент реально начинает уважать `step_windows` из Фазы 2 (иначе будем резать честный трафик, у которого просто нет данных для соблюдения ещё не отправленных окон).

**⚠️ Честное ограничение (2PAC, условие снятия Q9):** `seq` и `captured_at` — оба поля КОНТРОЛИРУЮТСЯ КЛИЕНТОМ (приходят в теле запроса, сервис им доверяет как заявленным данным). Order-by-evidence (6.1) + `captured_at`/timing-валидация (6.2/6.3) защищают от "ленивого" replay (заново поданного без учёта порядка/окна payload'а), но НЕ от атакующего, который целенаправленно конструирует payload с правильными `seq`/`captured_at`, зная эти правила. Для защиты от этого класса атаки нужен независимый серверный временной якорь (например `received_at`, проставляемый в момент фактического приёма запроса) — но все кадры сессии сегодня приходят ОДНИМ `POST /liveness/verdict`, поэтому межкадровых СЕРВЕРНЫХ таймингов физически не существует; `received_at` дал бы только одну точку на всю сессию, не per-frame якорь. Закрытие этого пробела — отдельный архитектурный вопрос (смена протокола на поштучную/потоковую отправку кадров или иной механизм), вне скоупа этого спринта. См. `docs/LIVENESS_CONTRACT_v1.md` §7.4.

**Зависимости:** 6.1 не зависит ни от чего, можно первым. 6.2 зависит от подтверждения партнёра, что шлёт `captured_at` стабильно. 6.3 зависит от Фазы 2 (существование `step_windows`) И от партнёра.

**Кто:** SNOOP (код), JAY-Z (если появится необходимость персистентного хранения аудита аномалий тайминга отдельно от текущего файлового audit_log — на этом этапе не ожидается, флаг пишет уже существующий audit-механизм).

**Риски:** 6.1 — риск ложных `STEP_NOT_DETECTED` на честном трафике, если реальная камера иногда даёт кадры не строго по порядку `seq` (маловероятно, `seq` присваивает клиент последовательно, но стоит подтвердить тестами на реальных сессиях перед снятием флага-предшественника). 6.2/6.3 — межкомандная зависимость, тот же риск, что и весь Rustam-контур: наш код готов раньше, чем партнёр реально начинает слать нужные данные.

## 7. Фаза 4 — Рамка `quality_certified`

**Явно НЕ придумываю числа.** Цели APCER/BPCER — предмет согласования Нурали + Рустам (см. §9), эта фаза проектирует только МЕХАНИЗМ, куда эти числа лягут, когда появятся.

### 7.1 Как измерять

APCER/BPCER для multi-frame `/liveness/*` пайплайна (Layer 0/2/3, НЕ то же самое, что уже измеренные Phase 1 `/pad/check` цифры, см. `docs/LIVENESS_CONTRACT_v1.md:211`) считаются офлайн, отдельным скриптом (`datasets/scripts/`, по аналогии с существующей тулзой RZA), прогоняющим реальный корпус (`bona_fide/real` + `print_attack/real` + `screen_photo/real` + `screen_video_replay/real` из `DATASET_COLLECTION_PROTOCOL_v1.md`) через реальный `/liveness/verdict` вызов (staging-стенд, не мок). Результат — отчёт `docs/plans/calibration/challenge_entropy_v1/CALIBRATION_REPORT.md` с числами APCER/BPCER на дату/коммит/`model_version`.

### 7.2 Где хранить цели

`app/config.py`, новые константы-плейсхолдеры (значение временно `None`/не установлено до согласования, НЕ произвольная цифра "для галочки"):
```
LIVENESS_TARGET_APCER: Optional[float] = None  # согласование Нурали+Рустам
LIVENESS_TARGET_BPCER: Optional[float] = None  # согласование Нурали+Рустам
LIVENESS_QUALITY_CERTIFIED: bool = False        # ручной флаг, не авто-вычисляемый
```

### 7.3 Как статус связывается с достижением целей

`LIVENESS_QUALITY_CERTIFIED` — **не автоматически вычисляемое** поле (в отличие от, скажем, порога EAR) — это ручной sign-off после того, как измеренные APCER/BPCER из `CALIBRATION_REPORT.md` (§7.1) пройдены против `LIVENESS_TARGET_APCER`/`LIVENESS_TARGET_BPCER` (§7.2) и явно подтверждены владельцем. Флаг переключает СЕМАНТИКУ уже существующего `LIVENESS_MODEL_VERSION` — при `LIVENESS_QUALITY_CERTIFIED=True` версия модели должна совпадать с тем `model_version`, на котором был прогнан сертифицирующий отчёт (защита от "сертифицировали одну версию, задеплоили другую" — стартап-проверка логирует `WARNING`, если версии разошлись, по аналогии с уже существующим стартап-логированием пустого `SERVICE_TOKEN`).

**Кто:** RZA (замер), Нурали+Рустам (согласование целей и итоговый sign-off), SNOOP (механика флага/стартап-проверки).

**Зависимости:** требует реального датасета (Фаза 5 частично, но `TURN_LEFT`/`TURN_RIGHT`-only корпус уже можно начинать собирать параллельно — не блокирует остальные фазы спринта).

**Риски:** это единственная фаза, которая упирается в людей/железо (сбор датасета с живыми людьми), а не в код — реалистично самая долгая часть спринта, как уже отмечено в `DATASET_COLLECTION_PROTOCOL_v1.md` про корпус `screen_video_replay` ("самый долгий пункт").

## 8. Фаза 5 — Калибровка BLINK и новых шагов (волновая раскатка пула)

| Волна | Что добавляем в `LIVENESS_CHALLENGE_STEPS_POOL` | Что нужно перед добавлением | Готовность датасет-протокола |
|---|---|---|---|
| **0 (уже прод)** | `TURN_LEFT`, `TURN_RIGHT` | — | — |
| **1** | `+BLINK` | Реальный корпус открытый/закрытый глаз (несколько человек, реальное моргание, продовое освещение) — прямая рекомендация уже записана в `LIVENESS_CONTRACT_v1.md` §4 п.1 и §5 п.2, спринт её не придумывает заново, а исполняет | Протокол не описывает явно "моргание" как отдельный `pose`/`attack` — нужно минимальное расширение `DATASET_COLLECTION_PROTOCOL_v1.md` (короткое видео/серия кадров моргания на bona_fide-субъектах) |
| **1** | `+NOD_UP`, `+NOD_DOWN` | Калибровка `LIVENESS_PITCH_NOD_MIN_DEG` на реальных данных (тот же процесс, что для yaw) | **Уже заложено** — `pose` в протоколе (§3) включает `up`/`down` без изменений протокола |
| **2** | `+SMILE` | Верификация индексации рта на реальном фото (аналог верификации глаз) + калибровка `LIVENESS_MAR_SMILE_MIN` | Протокол НЕ покрывает экспрессии — нужна отдельная небольшая правка протокола (v1.1): добавить `pose=smile` или отдельное поле `expression` |

**Обоснование волн:** Волна 1 собрана из шагов, где либо калибровка уже частично начата (BLINK), либо инфраструктура сбора данных уже существует без изменений (NOD, за счёт заранее заложенных `up`/`down` в протоколе RZA) — минимальная дополнительная работа. Волна 2 (SMILE) требует новой категории данных и новой верификации индексации — сознательно отложена до подтверждения, что Волна 1 вообще проходит калибровку успешно (иначе можно откатиться на резервный кандидат MOVE_CLOSER/FARTHER из §4.1 без пересборки всего плана).

**Кто:** RZA (калибровка + верификация индексации), 50 CENT/RICK ROSS (если нужны субъекты/железо для сбора — уже описано в блокерах `DATASET_COLLECTION_PROTOCOL_v1.md` §9).

**Важно:** пул в `LIVENESS_CHALLENGE_STEPS_POOL` растёт СТРОГО по мере прохождения калибровки каждого шага, а не одним PR на 6 шагов сразу — ровно то же правило, что уже действует в этом репозитории для BLINK (`SUPPORTED_STEPS` шире `LIVENESS_CHALLENGE_STEPS_POOL`, код готов заранее, включение в прод — отдельное решение).

## 9. Что требует решения Нурали (и в п.2/3 — Рустама)

1. **Числовые цели `LIVENESS_TARGET_APCER`/`LIVENESS_TARGET_BPCER`** (§7.2) — без них `quality_certified` физически не может стать `True`, это прямое условие Рустама, не наш выбор.
2. **Диапазон окон тайминга** (`LIVENESS_STEP_DELAY_MIN_MS`/`_MAX_MS`, §5.3) — конкретные миллисекунды влияют на UX-бюджет (уже под риском по латентности, `LIVENESS_CONTRACT_v1.md` §4 п.3) — нужно решение с оглядкой на реальный CPU-инференс, не только на энтропию.
3. **Кто и когда — субъекты для сбора Волны 1/2 датасетов** (§8) — организационный вопрос, не инженерный, тот же паттерн блокера, что уже стоит открытым в `DATASET_COLLECTION_PROTOCOL_v1.md` §9.
4. **MOVE_CLOSER/FARTHER как резерв** (§4.1) — подтвердить, что это именно резерв на случай провала SMILE, а не параллельная работа (экономия скоупа спринта).
5. **Расширение `DATASET_COLLECTION_PROTOCOL_v1.md` до v1.1** (BLINK-моргание + SMILE-экспрессия, §8) — небольшая правка, но это отдельный документ RZA, не переписываю его здесь явочным порядком.

## 10. Сводная последовательность фаз и зависимости

```
Фаза 0 (secrets ГСЧ)         — независимая, первая
Фаза 1 (NOD_UP/DOWN/SMILE в SUPPORTED_STEPS, вне пула) — независимая от Фазы 0/2/3
Фаза 2 (k=3-4, step_windows) — зависит от Фазы 0 (тот же ГСЧ)
Фаза 3.1 (order-by-evidence) — независимая, можно катить сразу, без флага
Фаза 3.2 (captured_at soft→hard) — зависит от подтверждения партнёра (мост egaz-mobile)
Фаза 3.3 (timing soft→hard)  — зависит от Фазы 2 + партнёра
Фаза 4 (quality_certified механика) — независимая от 0-3, зависит от Фазы 5 по данным
Фаза 5 (калибровка, волнами)  — самая долгая, людям/железу, частично параллельна остальному
```

**Пул `LIVENESS_CHALLENGE_STEPS_POOL` фактически растёт 2 → 3 (BLINK) → 5 (+NOD×2) → 6 (+SMILE)**, синхронно с Фазой 5, НЕ одним шагом — это отдельное сознательное решение, см. §8.
